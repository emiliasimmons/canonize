Composable agent skills for computational modelers — structured evidence trails behind every modeling decision. Projects are pure data (markdown, frontmatter, raw files); the skills carry one stdlib Python script.

## Architecture pointers

- For the design intent, zone model, disclosure spine, and invariants: `redesign-spec.md`
- For the reader-facing overview: `README.md`
- For individual skill behavior: `skills/<skill-name>/SKILL.md`
- For how the system works and its script interface (compile, check, sequence), agent-facing: `skills/canon/SKILL.md`
- For the script internals: `skills/canon/canon.py` and `skills/canon/README.md`
- For the LLM Wiki substrate pattern: `references/llm-wiki.md`
- For the OKF format the projects conform to: `references/knowledge-catalog/okf/SPEC.md`

## Code conventions

Each skill lives in `skills/<name>/` with a `SKILL.md` and, where relevant, format docs. Names describe what you are doing, not which system they belong to.

Format docs are the per-type conformance contracts and all live beside the single writer, `record-doc`:

```
skills/record-doc/
├── SKILL.md
├── decision-format.md
├── finding-format.md
├── source-format.md
├── concept-format.md
├── topic-format.md
├── register-format.md
├── glossary-format.md
├── provenance-format.md   # stock optional
└── trace-format.md        # stock optional
```

`skills/canon/` holds both the `canon` skill (`SKILL.md`, the knowledge layer most skills load for how the system works and how to run the script) and the stdlib `canon.py` script it documents. The script is stdlib only, invoked by skills against a project's substrate root; a project never needs it.

Skills compose rather than repeat: `grilling` is the shared interrogation subroutine, `canon` is the shared knowledge layer. User-facing front doors (`grill-with-docs`, `setup-canon`) set `disable-model-invocation`; subroutines (`grilling`, `canon`, `record-doc`) stay model-invokable. `export-bundles` is a stub — its banked design sits in `SKILL_STUB.md`, not loaded as a skill.

## Boundaries

- **Always**: exactly one skill writes pages — `record-doc`. Workflow skills decide what is worth writing and delegate the write.
- **Always**: every compiled surface (taxonomy, state, hub members, registers, per-zone indexes) is generated by `canon`, never hand-edited. Authored prose around a compiled block is never regenerated.
- **Always**: evidence is append-only — append, re-run, or supersede; never quietly rewrite. Wiki content must trace back to sources or evidence; if the backing material is removed, the claim is invalid.
- **Always**: links are root-anchored from the bundle root (`/evidence/...`), never file-relative.
- **Ask first**: structural taxonomy changes (new topic, new type, split/merge/rename). Tag minting is unilateral but registered in the same write.
- **Ask first**: changes to a project's `schema.md` or the `## docs` steering block; model code is never modified without per-diff approval.
- **Never** edit files under `references/` directly → re-vendor (replace the local copy) or add new reference files at `references/` root.

## Gotchas

- `references/` is **gitignored** vendor/reference material (`.gitignore` is literally `references`; no submodule). Anything built under `references/` is invisible to git, so skill tooling ships inside `skills/` (e.g. `build-docs-view/graph.py` is self-contained rather than depending on the vendored OKF viewer).
- Storage is zone-first; navigation is topic-first. Three zones, three mutabilities: `sources/` immutable and human-managed, `evidence/` immutable (append/supersede only), `wiki/` mutable and built incrementally. Hubs live at `wiki/topics/<name>.md` as typed sibling pages of their member directory `wiki/topics/<name>/`.
- Topic membership is by tag: a page tagged with a topic name appears in that hub. Evidence has no single-parent constraint; a decision appears in every hub it is tagged to.
- Parameter lifecycle stage (prior/calibrated) is **read from evidence**, never set by hand — the wiki compiles it. DR sequence numbers come from `canon sequence`, never picked by hand.
- There is no `log.md`; git is the log, one commit per logical write with a structured message (`ingest:`, `record: DR-NNNN`, `maintain:`).
- `audit-code` is the only skill that reads model code, strictly read-only: it reports a discrepancy table, then resolution flows through approved diffs or `record-doc`.
